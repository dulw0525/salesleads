# 客户线索分享模块需求说明书

## 1. 模块概述

### 1.1 产品定位
聚流App - 专为保险代理人打造的客户经营与展业互助平台

### 1.2 模块名称
客户线索分享模块（供给侧）

### 1.3 模块目标
- 将拥有溢出客户资源或跨地域资源的代理人线索合规流转至平台
- 通过标准化流程确保线索质量与合规性
- 建立公平的激励机制，鼓励优质线索分享

### 1.4 设计原则
- **业务合规化**：剥离"数据买卖"属性，重塑为基于积分的"线索分享与经营互助"
- **财务合规化**：充值与提现严格解耦，提现资金定性为"劳务推广报酬"
- **治理自动化**：依赖算法模型与信誉分体系实现平台生态自治

## 2. 功能详细说明

### 2.1 客户画像录入

#### 2.1.1 表单字段设计

| 字段名称 | 数据类型 | 必填 | 说明 |
|---------|---------|------|------|
| 客户姓名 | 文本 | 是 | 真实姓名 |
| 手机号码 | 文本 | 是 | 11位手机号，系统自动格式化 |
| 年龄段 | 单选 | 是 | 18-25, 26-35, 36-45, 46-55, 56+ |
| 性别 | 单选 | 是 | 男/女 |
| 职业类型 | 下拉选择 | 是 | 企业主、白领、公务员、教师、医生、其他 |
| 意向险种 | 多选 | 是 | 车险、寿险、意外险、健康险、财产险 |
| 保费预算区间 | 单选 | 是 | <1000元, 1000-5000元, 5000-10000元, >10000元 |
| 所在城市 | 级联选择 | 是 | 省-市-区三级选择 |
| 备注说明 | 文本域 | 否 | 其他补充信息，限200字 |

#### 2.1.2 前端交互要求
- **实时校验**：手机号格式实时验证，不符合格式时即时提示
- **防重复录入**：提交前自动检测该手机号是否已在系统中存在（基于Hash值比对）
- **草稿自动保存**：表单填写过程中，每30秒自动保存草稿
- **字段引导**：每个字段下方提供简短的填写示例或说明

### 2.2 合规前置检查

#### 2.2.1 法律承诺书
在提交前，必须强制用户勾选以下承诺：

```
《信息来源合法及已获授权承诺函》

本人郑重承诺：
1. 本线索信息来源合法，不涉及侵犯他人隐私或非法获取
2. 已获得客户本人明确授权，允许在保险展业场景中使用本信息
3. 所提供信息真实有效，如有虚假愿承担相应法律责任
4. 承诺不将平台线索用于任何违法违规用途

签署人：[自动填充用户名]
签署日期：[自动填充提交时间]
```

#### 2.2.2 界面要求
- 承诺函内容必须完整展示，不可折叠
- 勾选框位于承诺函正下方，且必须勾选后才能点击提交按钮
- 提交按钮默认禁用，勾选后方可点击
- 勾选时弹出温馨提示："请确保您已真实获得客户授权，虚假承诺将影响您的信誉分"

### 2.3 自动化预检流程

#### 2.3.1 系统查重逻辑
**触发时机**：用户点击提交按钮后立即触发

**查重流程**：
1. 对手机号进行SHA256哈希计算
2. 查询系统线索表中是否存在相同哈希值
3. 如果存在，进一步比对：
   - 相同手机号在7天内是否已被分享
   - 如果已被分享且状态为"已申领"或"已过期"，则判定为重复

**处理结果**：
- **重复检测通过**：继续下一步校验
- **重复检测失败**：立即终止流程，返回错误提示："该客户已在近期被分享，请确认是否为重复录入"

#### 2.3.2 第三方API校验

**校验内容**：
1. **空号检测**：调用运营商接口验证号码是否为空号
2. **停机检测**：验证号码是否处于停机状态
3. **实名验证**：验证号码是否已完成实名认证

**接口调用策略**：
- **超时设置**：每个API调用超时时间为3秒
- **失败处理**：若某个接口调用失败，记录日志但不影响后续流程
- **结果权重**：空号检测为硬性校验，停机和实名校验为参考性校验

**校验结果判定**：
- **通过条件**：非空号 + 未停机
- **不通过条件**：空号或停机状态
- **特殊情况**：实名未通过但其他两项通过，标记为"待人工复核"

#### 2.3.3 预检结果反馈

| 结果类型 | 前端提示 | 后端处理 |
|---------|---------|---------|
| 预检通过 | "线索预检通过，正在分配激励积分..." | 继续流程，进入激励预分配 |
| 空号检测失败 | "手机号为空号，请核实后重新提交" | 终止流程，不保存记录 |
| 停机检测失败 | "手机号已停机，请核实后重新提交" | 终止流程，不保存记录 |
| 系统查重失败 | "该客户已在近期被分享" | 终止流程，不保存记录 |
| API调用超时 | "网络异常，请稍后重试" | 终止流程，允许重新提交 |

### 2.4 激励预分配机制

#### 2.4.1 积分价值评估模型

**基础分值设定**：
- 完整画像线索：100积分基础值
- 缺失1-2项非关键信息：80积分
- 仅提供基本联系方式：50积分

**加分项**：
| 加分维度 | 加分值 | 说明 |
|---------|--------|------|
| 保费预算 > 10000元 | +30积分 | 高价值客户 |
| 意向险种包含寿险/健康险 | +20积分 | 长期价值客户 |
| 所在城市为一线城市 | +15积分 | 高消费能力地区 |
| 职业类型为高净值职业 | +10积分 | 企业主、高级管理层等 |

**扣分项**：
| 扣分维度 | 扣分值 | 说明 |
|---------|--------|------|
| 使用虚拟运营商号段 | -20积分 | 信用风险较高 |
| 实名验证未通过 | -10积分 | 需人工复核 |
| 非工作时间段提交 | -5积分 | 22:00-6:00期间 |

**最终积分计算公式**：
```
最终积分 = 基础分值 + Σ(加分项) - Σ(扣分项)
最低积分：50分
最高积分：200分
```

#### 2.4.2 冻结状态管理

**冻结规则**：
- 预分配积分状态标记为"冻结中"（status: frozen）
- 冻结期：自分享成功起30天
- 冻结期间积分不可提现、不可消费

**解冻条件**：
1. **自动解冻**：30天后自动转为可用状态（仅限未被投诉线索）
2. **提前解冻**：被申领人确认线索"真实有效"后立即解冻
3. **强制解冻**：仲裁判定分享人无责后解冻

**状态流转**：
```
冻结中 → 可用（正常解冻/申领确认）
冻结中 → 扣除（投诉属实）
冻结中 → 退还（仲裁撤销）
```

#### 2.4.3 积分类型区分

**数据库设计**：
```sql
积分记录表 (points_record)
- id: 主键
- user_id: 用户ID
- amount: 积分数量
- type: 积分类型 (recharge=充值, incentive=激励)
- status: 状态 (available=可用, frozen=冻结, used=已使用)
- source: 来源 (share=分享, recharge=充值, task=任务)
- related_id: 关联线索ID
- created_at: 创建时间
- updated_at: 更新时间
```

**提现权限**：
- 只有`type = incentive`且`status = available`的积分可以提现
- `type = recharge`的积分永久不可提现
- 提现时系统自动过滤不可提现积分

### 2.5 提交流程状态机

#### 2.5.1 状态定义

```
DRAFT (草稿)
  ↓ [开始填写]
EDITING (编辑中)
  ↓ [保存草稿]
DRAFT (草稿)
  ↓ [提交]
VALIDATING (校验中)
  ↓ [校验通过]
PENDING_APPROVAL (待审核)
  ↓ [系统审核通过]
PUBLISHED (已发布)
  ↓ [被申领/过期]
COMPLETED (已完成)

异常流程：
VALIDATING → FAILED (校验失败)
PENDING_APPROVAL → REJECTED (审核拒绝)
```

#### 2.5.2 状态转换条件

| 当前状态 | 触发动作 | 目标状态 | 必要条件 |
|---------|---------|---------|---------|
| DRAFT | 开始编辑 | EDITING | 用户进入编辑页 |
| EDITING | 保存草稿 | DRAFT | 表单数据保存成功 |
| EDITING | 提交 | VALIDATING | 表单验证通过 + 承诺函已勾选 |
| VALIDATING | 预检通过 | PENDING_APPROVAL | 查重通过 + 第三方校验通过 |
| VALIDATING | 预检失败 | FAILED | 任一校验未通过 |
| PENDING_APPROVAL | 系统审核通过 | PUBLISHED | 无需人工干预，自动审核 |
| PENDING_APPROVAL | 需人工复核 | PENDING_MANUAL | 实名未通过等特殊情况 |
| PUBLISHED | 被申领 | COMPLETED | 有用户成功申领 |
| PUBLISHED | 过期 | EXPIRED | 发布后30天未被申领 |

### 2.6 异常处理机制

#### 2.6.1 网络异常处理

**场景1：提交过程中网络中断**
- 前端本地保存草稿
- 提示用户："网络连接异常，草稿已保存，建议检查网络后重试"
- 提供"重试提交"和"保存草稿"两个选项

**场景2：第三方API调用失败**
- 记录失败日志，包含失败原因和时间戳
- 继续执行流程，但标记为"待人工复核"
- 后台定时任务每5分钟重试一次，最多重试3次

**场景3：数据库写入失败**
- 立即回滚所有相关操作
- 提示用户："系统繁忙，请稍后重试"
- 记录详细错误日志供技术排查

#### 2.6.2 用户操作异常

**重复提交防护**：
- 提交按钮点击后立即禁用，防止重复提交
- 后端设置10秒防重提交机制
- 如果检测到重复提交，返回友好提示："正在处理中，请勿重复提交"

**表单数据异常**：
- 字段长度超限：前端实时提示，后端做最终校验
- 特殊字符过滤：自动过滤XSS攻击字符
- 敏感词检测：对备注字段进行敏感词过滤，发现敏感词时提示修改

#### 2.6.3 系统级异常

**积分发放失败**：
- 记录失败线索信息
- 进入补偿队列，定时任务每小时尝试补偿发放
- 3次补偿失败后，标记为异常状态，通知运营人工处理

**线索发布失败**：
- 保持积分冻结状态
- 记录失败原因
- 允许用户在"我的草稿"中重新提交

## 5. 业务规则详细说明

### 5.1 线索有效期规则

**基础规则**：
- 线索发布后有效期为30天
- 30天内未被申领，自动过期
- 过期后不可被申领，但数据保留用于统计

**延长规则**：
- 如果线索在25天时仍未被申领，系统发送提醒："您的线索即将过期，是否需要重新发布？"
- 用户可选择"重新发布"，生成新的线索记录，原记录标记为已过期

### 5.2 积分冻结与解冻规则

**冻结规则**：
```javascript
// 积分冻结逻辑
function freezePoints(leadId, points) {
  // 1. 创建冻结积分记录
  createPointsRecord({
    user_id: currentUser.id,
    amount: points,
    type: 'incentive',
    status: 'frozen',
    source: 'share',
    related_id: leadId
  });

  // 2. 设置冻结期30天
  setFreezeExpiry(leadId, 30 * 24 * 60 * 60 * 1000); // 30天毫秒数

  // 3. 记录冻结日志
  logFreezeAction(leadId, points);
}
```

**解冻规则**：
```javascript
// 积分解冻逻辑
function unfreezePoints(leadId, triggerReason) {
  // triggerReason: 'CLAIM_CONFIRM' | 'TIME_EXPIRE' | 'ARBITRATION_PASS'

  const lead = getLeadById(leadId);

  if (triggerReason === 'CLAIM_CONFIRM') {
    // 申领人确认真实有效，立即解冻
    updatePointsStatus(lead.related_points_id, 'available');
  } else if (triggerReason === 'TIME_EXPIRE') {
    // 30天到期，检查是否有投诉
    if (!hasComplaints(leadId)) {
      updatePointsStatus(lead.related_points_id, 'available');
    }
  } else if (triggerReason === 'ARBITRATION_PASS') {
    // 仲裁判定无责，立即解冻
    updatePointsStatus(lead.related_points_id, 'available');
  }
}
```

### 5.3 查重规则

**查重时间窗口**：
- 同一手机号30天内只能被分享一次
- 已被申领的线索，原分享人60天内不得重新分享同一客户

**查重算法**：
```python
def check_duplicate(phone):
    # 1. 计算手机号哈希
    phone_hash = sha256(phone)

    # 2. 查询最近30天的线索
    recent_leads = Lead.query.filter(
        Lead.phone_hash == phone_hash,
        Lead.status.in_([1, 2, 3]),  # 审核中、已发布、已申领
        Lead.created_at >= datetime.now() - timedelta(days=30)
    ).all()

    # 3. 判定是否重复
    if recent_leads:
        return {
            'is_duplicate': True,
            'reason': f'该客户在{len(recent_leads)}天前已被分享',
            'previous_lead_id': recent_leads[0].id
        }

    return {'is_duplicate': False}
```

### 5.4 信誉分影响规则

**分享成功加分**：
- 线索被申领后，申领人确认"真实有效"：+2分
- 单月分享成功线索超过10条：额外+5分

**分享失败扣分**：
- 被投诉且核实为虚假线索：-5分/次
- 被投诉且核实为空号/停机：-3分/次
- 单月被投诉次数超过3次：额外-10分

**信誉分等级**：
| 分数区间 | 等级 | 权益 |
|---------|------|------|
| 95-100 | S | 优先推荐，提现秒到账 |
| 90-94 | A | 正常权益 |
| 80-89 | B | 限流，提现人工审核 |
| 60-79 | C | 严重限制，冻结权限7天 |
| <60 | D | 永久封禁 |

## 8. 性能与安全要求

### 8.1 性能指标

| 指标项 | 目标值 | 说明 |
|-------|--------|------|
| 表单提交响应时间 | <2秒 | 从点击提交到返回结果 |
| 手机号查重响应时间 | <500ms | 数据库查询时间 |
| 第三方API调用超时 | 3秒 | 单个API调用 |
| 并发支持 | 1000+ | 同时提交线索的用户数 |

### 8.2 安全要求

**数据加密**：
- 手机号：使用AES-256对称加密存储
- 姓名：使用AES-256对称加密存储
- 传输过程：全程HTTPS

**隐私保护**：
- 前端展示：手机号中间4位脱敏显示
- 数据库存储：原始数据加密，仅系统可解密
- 访问权限：严格控制数据访问权限，仅相关业务模块可访问

**防刷机制**：
- 提交频率限制：同一用户每小时最多提交10条线索
- IP限制：同一IP地址每小时最多提交50条线索
- 验证码：高频操作时触发图形验证码
